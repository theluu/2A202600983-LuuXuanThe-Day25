So sánh: Có cache vs Không cache

Chạy live 2 lần cùng bộ query (40 request/lần) — một lần cache.enabled=true, một lần false. Cùng provider (primary fail 25%).

Kỹ thuật cache đang dùng

  • Semantic similarity = Cosine trên character 3-gram + word token. Mỗi câu tách thành từ + các 3-ký-tự (vd "hello"→hel,ell,llo), đếm tần suất thành vector, tính cosine. Mịn hơn Jaccard (đo trùng token thô).
  • Privacy guardrail — regex chặn query chứa balance / password / ssn / user-<số> / account-<số>: không lưu, không trả từ cache.
  • False-hit guardrail — nếu 2 câu chứa số 4 chữ số khác nhau (vd năm 2024≠2026) thì từ chối dù điểm cao, ghi vào false_hit_log.
  • TTL 300s + ngưỡng similarity 0.92 (test cho thấy 0.85 gây false-hit).

Vai trò của Redis

  • Vấn đề: cache in-memory nằm trong RAM từng process. Có N instance sau load balancer → query cache ở instance A là MISS ở B…N, hit rate tụt còn ~1/N; mất sạch khi restart/deploy.
  • Giải pháp: SharedRedisCache lưu chung 1 keyspace Redis (Hash rl:cache:<md5> gồm query/response, TTL qua EXPIRE). Mọi instance thấy chung → hit rate scale theo toàn fleet, sống sót qua deploy.
  • Cùng guardrail: privacy + false-hit áp dụng y hệt bản in-memory.
