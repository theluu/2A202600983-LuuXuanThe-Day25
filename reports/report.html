01Yêu cầu bài toán

Repo starter cung cấp kiến trúc, interface và bộ test; nhiệm vụ là tự implement toàn bộ logic reliability để biến 25 test đang FAIL → 0 fail, chấm trên thang 100 điểm dựa vào bằng chứng (evidence), không phải mô tả suông.

Bài toán cốt lõi

Một agent gọi LLM qua provider bên ngoài sẽ gặp: độ trễ thất thường, lỗi tạm thời, chi phí theo token, và provider sập. Nếu cứ retry mù sẽ tạo "retry storm" đánh sập luôn provider đang yếu. Cần một lớp trung gian đảm bảo: người dùng không bao giờ thấy lỗi cứng, chi phí được kiểm soát, và hệ thống tự phục hồi khi provider khỏe lại.

02Cách xử lý

Triển khai test-driven theo đúng thứ tự phụ thuộc: mỗi lớp dựa trên lớp trước, chạy test cho xanh rồi mới sang phase kế.

Kiến trúc luồng request

User Request │ ▼ [Gateway.complete()] │ ▼ [Cache: memory | Redis] ──HIT──▶ trả GatewayResponse(route="cache_hit:<score>", cost=0) │ MISS / uncacheable ▼ [Circuit Breaker: primary] ──call──▶ Provider "primary" │ OPEN? fail fast (CircuitOpenError) │ success ▶ cache.set(), route="primary" │ ProviderError ▶ record_failure, continue│ ▼ [Circuit Breaker: backup] ──call──▶ Provider "backup" │ OPEN? fail fast │ success ▶ cache.set(), route="fallback" │ ProviderError ▶ record_failure, continue│ ▼ [Static fallback] route="static_fallback", error=<last error> "The service is temporarily degraded. Please try again soon."

Chi tiết từng phase

1

Circuit Breaker — máy trạng thái 3 pha

CLOSED (đếm lỗi) → OPEN (fail-fast tới khi hết reset_timeout) → HALF_OPEN (1 probe) → CLOSED/OPEN. Điểm mấu chốt: record_failure() dùng if/elif tách 2 lý do probe_failure (HALF_OPEN sập) và failure_threshold_reached — tránh retry storm và ghi transition log minh bạch.

2

Semantic Cache + Guardrail

similarity() = cosine trên char-3gram + word token (mịn hơn Jaccard). Privacy guard: không lưu/trả query chứa balance/password/SSN/user-id. False-hit guard: từ chối match nếu query & key chứa số 4 chữ số khác nhau (vd năm 2024≠2026) và ghi vào false_hit_log.

3

Gateway pipeline

complete(): kiểm cache → duyệt chuỗi provider qua breaker (route primary/fallback) → nếu tất cả sập trả static fallback kèm error. Bắt cả ProviderError lẫn CircuitOpenError để chuyển provider kế.

4

Chaos + Metrics

run_scenario() chạy N request, gom latency/cost/cache/fallback; calculate_recovery_time_ms() quét transition log (open→closed). Tiêu chí pass/fail riêng cho từng scenario. write_csv() xuất metrics phẳng.

5

Shared Redis Cache

SharedRedisCache lưu Hash rl:cache:<md5> gồm query/response, TTL bằng EXPIRE phía server. Lookup: exact key trước, rồi scan_iter + similarity. Mọi instance dùng chung keyspace → hit rate scale theo toàn fleet.

6

Report

Báo cáo đầy đủ 9 section: diagram, bảng config có lý do, SLO, metrics, so sánh cache on/off, bằng chứng Redis (KEYS + shared state), failure analysis.

Pattern reliability áp dụng

  • Circuit Breaker — cô lập provider lỗi, fail-fast, tự phục hồi qua probe
  • Fallback chain + Graceful degradation — primary → backup → static message
  • Semantic caching — giảm tải & chi phí, có guardrail bảo mật/đúng ngữ nghĩa
  • Chaos engineering — bơm lỗi có chủ đích để kiểm chứng hành vi dưới áp lực
  • SLO/observability — P50/P95/P99, availability, recovery time đo được

05Ý nghĩa

Vì sao những con số trên quan trọng với một hệ thống agent chạy thật.

🛡️ Độ tin cậy

Dù provider chính sập 100%, người dùng vẫn nhận câu trả lời (qua backup/cache) hoặc thông điệp degraded lịch sự — không bao giờ lỗi trắng. Circuit breaker chặn retry storm và hệ thống tự phục hồi trong ~2.3 giây sau khi provider khỏe lại.

💰 Chi phí

Cache cắt 63% chi phí token ở mức memory và 69% ở mức Redis. Với LLM tính tiền theo token, đây là khoản tiết kiệm tuyến tính theo lưu lượng — càng nhiều traffic lặp lại, càng lời.

🌐 Khả năng mở rộng

Cache in-memory suy biến về 1/N hit rate khi có N replica và mất sạch khi restart. Redis shared cache giúp hit rate scale theo toàn fleet và sống sót qua deploy — điều kiện bắt buộc để chạy nhiều instance.

🔒 An toàn & đúng ngữ nghĩa

Privacy guardrail đảm bảo dữ liệu nhạy cảm (số dư, mật khẩu, user-id) không bao giờ bị cache. False-hit guard tránh trả nhầm chính sách năm 2024 cho câu hỏi năm 2026 — lỗi âm thầm nguy hiểm nhất của semantic cache.

Điểm yếu còn lại & hướng khắc phục: trạng thái circuit breaker hiện nằm trong RAM từng process — mỗi replica tự học "primary đang sập" nên toàn fleet vẫn có thể phát sinh retry storm nhỏ. Hướng đi production: lưu counter/state breaker trong Redis (INCR+EXPIRE) để mọi instance mở/phục hồi đồng bộ, và thêm graceful degradation khi Redis chết thì rơi về cache in-memory.

Kết luận

Lab chứng minh 5 pattern reliability kinh điển — circuit breaker, fallback chain, semantic caching, chaos testing, SLO/observability — kết hợp lại biến một chuỗi gọi LLM mong manh thành dịch vụ 98.7% khả dụng, chi phí giảm 65%, tự phục hồi <2.5s, tất cả được kiểm chứng bằng test tự động và metrics tái lập được, không phải bằng lời.
